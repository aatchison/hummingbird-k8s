# Phase 1B cycle 4: wait_apiserver_back gate parity vs bash twin
#
# Fixture naming convention (locked in PR #325): cycle<N>_<block>.txt.
#
# Captured 2026-05-27 against the live geary cluster (hbird-cp1 +
# hbird-w1 + hbird-w2; CP_IP=192.168.122.212). Cycle 4 (#329) wires:
#
#   - wait_apiserver_back               (bash scripts/update-cluster.sh:1077)
#
# The helper gates the post-reboot recovery path: after the CP reboots
# the apiserver is unreachable for 60-120s. wait_apiserver_back polls
# `kubectl get --raw=/readyz` via the CP every 10s until the endpoint
# answers `ok` (or until APISERVER_TIMEOUT=300s elapses, the bash
# twin's outer fail wording). The /readyz endpoint is chosen
# deliberately over `get nodes` because the apiserver answers /readyz
# BEFORE the etcd watcher has fully caught up — `get nodes` can return
# stale 5xx for an extra 5-30s past the point where the apiserver is
# "answering" from a control-flow perspective. The cycle 4 wiring
# preserves that bash behavior verbatim.
#
# === Important note: steady-state validation only ===
#
# This fixture captures a STEADY-STATE happy-path probe rather than a
# full CP-reboot cycle. Rationale, from #329:
#
#   "operator must be on tmux 5:0 for the apiserver-flap window —
#    kubectl is unreachable for 60-120s mid-cycle, can't be
#    unattended."
#
# A real CP reboot would tear down the apiserver across the recovery
# window, blocking every downstream consumer of this cluster (verify-*
# suites, parallel agent dispatches, future cycle live tests, etc.).
# The high-risk operation requires explicit operator standby that the
# Phase 1B cycle-by-cycle automation isn't built to support
# unattended.
#
# What this fixture DOES validate:
#
#   1. The /readyz endpoint exists + returns `ok` on the steady-state
#      cluster — pins the bash twin's chosen probe target.
#   2. The SSH+kubectl wiring through the cp_kubectl shim (root@CP_IP
#      via ProxyJump=KVM_HOST) correctly delivers the kubectl call
#      to the apiserver and surfaces the body.
#   3. The polling-loop entry condition is the IDENTICAL command both
#      sides use — bash twin line 1088 + Rust wait_apiserver_back.
#
# What this fixture does NOT validate:
#
#   1. The polling-loop iteration during a real apiserver-down window
#      (60-120s of kubectl Err → kubectl Ok at the recovery edge).
#      The cycle 2 (`wait_ssh_back`) + cycle 3 (`wait_node_ready`)
#      live tests already pin the IDENTICAL polling pattern (Err ⇒
#      re-iterate, Ok ⇒ return-with-elapsed-s) against transient
#      recovery windows — cycle 4 reuses that exact shape, just
#      against `/readyz` instead of an SSH probe / kubectl get node.
#   2. The 300s outer timeout. That's exercised by the bash twin's
#      production callers (update_cp at line 1238) and inherited via
#      the dry-run fixture's `timeout 300s` log line.
#
# The polling-loop failure path (timeout-on-apiserver-down) is
# unit-test territory rather than live-test, because (a) we can't
# safely tear down the live apiserver, and (b) the loop's polling
# semantics are pinned in unit tests for the sibling helpers.
#
# Both the bash and Rust sides drove the IDENTICAL SSH chain through
# the IDENTICAL kubectl operation:
#
#   bash side: `ssh -J $KVM_HOST root@$CP_IP "kubectl --kubeconfig=
#              /etc/kubernetes/admin.conf get --raw=/readyz"`
#              (operator equivalent of the bash twin's cp_kubectl shim
#              at scripts/update-cluster.sh:425 invoking the body of
#              wait_apiserver_back at line 1088)
#   rust side: tests/cycle4_apiserver_live.rs invoking
#              hbird_ssh::Client::run with SshOptions::new(cp_ip)
#                .with_user("root")
#                .with_proxy_jump(kvm_host)
#              — same shape as cycles 1/2/3 cp_kubectl path.
#
# Both runs targeted hbird-cp1 (per #329's coordination note: cycle 4
# is the CP-targeted cycle). Both runs ended with the cluster at 3/3
# Ready.
