=== verdict: PARITY (chain semantics confirmed) ===
- Three verifiers run in order encryption → hardening → app-deploy,
  matching `make verify-all` chain order.
- All three PASS sequentially; chain exits 0 only when every link
  passes (Rust uses `?` propagation, same first-failure semantics as
  bash's Make chain).
- Cluster left 3/3 Ready; smoketest namespace cleaned up.
- Tracing ERROR line in check 1/3 is the same hbird-ssh
  `err(Debug)` instrument cosmetic noise documented in
  cycle_verify_hardening.txt.

=== bash counterpart ===
`make verify-all` was NOT re-captured for this composite fixture
because:
  (a) Bash's verify-hardening still hits the kc-wrapper stdin bug
      documented in cycle_verify_hardening.txt and would short-circuit
      the chain with FAIL on check 1/3.
  (b) The chain semantics ("each target runs only if the previous
      one returned 0") are a Make-recipe property, not a bash-twin
      property — Make's first-failure behavior is well-defined and
      well-tested in CI.
The individual-verifier fixtures (cycle_verify_encryption.txt,
cycle_verify_hardening.txt, cycle_verify_app_deploy.txt) cover the
bash-vs-Rust diff per verifier; this fixture proves the Rust chain
ties them together with matching exit-on-failure semantics.
